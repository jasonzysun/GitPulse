# 支持批量生成工作报告

> GitHub Issue: https://github.com/GoldenZqqq/GitPulse/issues/1
> 提出者: jasonzysun

## Goal

允许用户一次性生成多份报告（如一个月的30份日报、4份周报），避免逐份手动操作。覆盖补历史记录、批量归档等真实场景。

## Background

当前 GitPulse 每次交互只能生成一份报告。用户如果需要补一个月的日报，需要手动切换日期30次。批量生成功能将复用现有的 commit 提取、报告渲染、格式导出基建，在外层加一个"日期拆分 + 循环生成 + 进度反馈"的编排层。

## Requirements

### Phase 1 (MVP)

- 用户选择一个时间范围（start_date ~ end_date）和拆分粒度（按天/按周/按月）
- 系统自动将时间范围拆分为 N 个子周期
- 对每个子周期调用现有报告生成流程，生成对应报告
- 支持选择导出格式（Markdown / DOCX / PDF），每份报告导出为独立文件
- 用户选择输出目录
- 文件命名规则：`{日期范围}-{报告类型}.{ext}`（如 `2026-07-01-日报.md`、`2026-W28-周报.docx`）
- 生成过程中显示进度（当前第 N/M 份）
- 生成完成后显示成功/失败汇总，允许打开输出目录
- 批量生成复用现有筛选条件（作者、仓库、分支）

### Phase 2A（当前增量）

- 用户可以同时选择 Markdown / Word / PDF 中的一种或多种格式
- 拆分粒度支持“整段范围”，将所选起止日期作为一份自定义周期报告
- 每个子周期只生成一次报告内容，再复用内容写出所有选中格式
- 用户可以配置文件名模板，默认 `{period}-{type}.{ext}`，保持 MVP 文件名兼容
- 支持 `{period}`、`{date}`、`{week}`、`{month}`、`{startDate}`、`{endDate}`、`{author}`、`{project}`、`{type}`、`{ext}` 变量
- 未知变量、空模板、缺少 `{ext}`、非法文件名与重复文件名必须由后端统一处理，不能覆盖已有批次文件
- 进度、成功数、失败数和总数按实际输出文件数量统计

### Phase 2B（已完成）

- 分组方式支持“全部汇总 / 按作者 / 按项目”，默认保持全部汇总
- 按作者分组时，使用作者别名归并后的展示名作为分组身份
- 按项目分组时，优先使用现有项目映射，未映射时回退到 `仓库(分支)`
- 分组集合从整个所选时间范围内的提交确定，然后为每个“周期 × 分组”生成报告；分组在某个周期无提交时仍生成空报告
- `{author}` / `{project}` 在分组模式下替换为当前分组名
- 365 上限按 `周期数 × 分组数 × 格式数` 的实际输出文件数计算
- 整个时间范围没有可用于作者/项目分组的提交时，返回明确错误而不是生成无法识别归属的文件

### Phase 3（后续评估）

- 可选 AI 润色（批量场景下需考虑 API 限流和费用）

## Constraints

- 批量生成期间不阻塞 UI，使用 Tauri event 推送进度
- 单次批量最多生成 365 个实际输出文件，防止多格式组合造成意外的大批量写入
- 复用现有 `commit_pipeline`、`report`、`docx`、`pdf` 模块，不重复造轮子
- 文件写入失败不中断整个批量流程，记录失败项并最终汇总

## Acceptance Criteria

- [x] 用户可在前端选择时间范围 + 拆分粒度 + 导出格式 + 输出目录
- [x] 后端按拆分粒度将时间范围切分为子周期，逐份生成报告
- [x] 每份报告内容与单独生成该周期报告的结果一致
- [x] 生成过程中前端显示实时进度
- [x] 生成完成后显示成功/失败汇总
- [x] 单份失败不影响后续报告生成
- [x] 文件命名包含日期和报告类型，无冲突
- [x] 支持 Markdown、DOCX、PDF 三种格式
- [x] 支持同时选择并导出多种格式，且每个周期只生成一次内容
- [x] 自定义命名模板可预期地替换变量、清理非法字符并避免文件覆盖
- [x] 进度与汇总数量等于实际尝试写出的文件数量
- [x] 支持全部汇总、按作者和按项目三种分组方式
- [x] 作者别名与项目映射在分组和文件命名中保持一致
- [x] 分组模式的进度、汇总和 365 上限按实际输出文件数计算
- [x] 支持将整个所选日期范围生成一份自定义周期报告
- [x] 文件名模板变量可点击复制，并显示成功或失败提示
