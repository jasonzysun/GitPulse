# Trellis + Grill Me Workflow

This project uses Trellis as the persistent AI workflow layer and `grill-with-docs` as the planning pressure-test habit. The goal is not ceremony; it is to make Claude Code and Codex remember the same project language, constraints, and task decisions through files.

## What Was Initialized

- `.trellis/`: workflow, task, spec, and workspace memory system.
- `.claude/`: Claude Code commands, hooks, agents, and Trellis skills.
- `.codex/`: Codex agents and prompt-submission hook configuration.
- `.agents/`: shared agent assets generated by Trellis for tools that read that convention.
- `CONTEXT.md`: grill me glossary for GitPulse domain language.
- `.trellis/spec/`: project-specific frontend and Tauri/Rust guidelines.

## Daily Use

For a small one-off change, it is fine to skip Trellis and work normally. For anything with unclear requirements, cross-layer behavior, public UX, report semantics, release flow, or security implications, create a Trellis task first.

```powershell
$ErrorActionPreference = 'Stop'
python .\.trellis\scripts\task.py create "Describe the task" --slug short-name
```

During planning, ask the agent to "grill me". The expected behavior is:

1. Check code and docs before asking questions.
2. Ask one question at a time.
3. Recommend an answer for each question.
4. Update `CONTEXT.md` immediately when product language is clarified.
5. Write requirements into the active task's `prd.md`.
6. Use `design.md` and `implement.md` only when the task is complex enough to need them.

After the plan is reviewed, start implementation:

```powershell
$ErrorActionPreference = 'Stop'
python .\.trellis\scripts\task.py start <task-dir>
```

For Codex, this repository is configured for inline mode, so the main Codex session should read the Trellis task artifacts and edit directly. For Claude Code, use the generated `/trellis:*` commands and Trellis skills when available.

## Grill Me Prompts

Use these prompts when you want the workflow explicitly:

```text
Use Trellis for this. Create a task, then grill me before implementation.
```

```text
Grill this plan against GitPulse's CONTEXT.md and Trellis spec. Ask one question at a time and recommend your answer.
```

```text
Before coding, check whether any term in this plan conflicts with CONTEXT.md. Update the glossary if we settle a new term.
```

## When To Create an ADR

Do not create ADRs for ordinary task plans. Create one only when a decision is hard to reverse, surprising without context, and based on a real trade-off. If all three are not true, capture the outcome in the Trellis task or spec instead.

## Verification Habit

- Frontend changes: `npm run build`; targeted Playwright when UI flows change.
- Rust changes: `cd src-tauri && cargo check && cargo test`.
- Release changes: follow `CONTRIBUTING.md` and the `gitpulse-release-publisher` skill.
- Workflow/spec changes: run `python .\.trellis\scripts\get_context.py --mode packages` and inspect the affected spec files.

## Codex Hook Note

Trellis generated `.codex/hooks.json`. Your global Codex config already has `features.hooks = true`; in a fresh Codex install, run `/hooks` once and approve the Trellis `UserPromptSubmit` hook so workflow breadcrumbs can be injected automatically.
