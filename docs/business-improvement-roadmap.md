# GitPulse 业务体验改进路线图

## 背景

GitPulse 当前已经能完成多仓库扫描、日报/周报/自定义区间/月报生成、AI 润色、导出和诊断。下一阶段的业务重点不应只是增加按钮，而是减少用户在真实交付报告时的判断成本：为什么没有结果、报告够不够能交付、多人/多身份怎么汇总、项目名怎样更贴近业务语境。

## 优先级说明

- P0：高频阻塞，直接影响用户能否顺利生成报告。
- P1：明显提升报告可交付性或团队场景覆盖。
- P2：增强专业度、复用性或长期留存。

## 值得推进的业务任务

| ID | 优先级 | 状态 | 用户问题 | 建议改进 | 验收标准 |
| --- | --- | --- | --- | --- | --- |
| BIZ-01 | P0 | 已完成 | 生成空报告时，用户不知道是没有提交、作者不匹配、仓库没扫到，还是时间范围选错。 | 在空报告和警告区域给出可行动原因与下一步建议。 | 当提交数为 0 时，界面能说明当前筛选范围，并提示检查作者、日期、分支范围或仓库索引。 |
| BIZ-02 | P0 | 已完成 | 空作者现在代表“全部作者”，但界面仍容易让用户误解为配置缺失。 | 将主界面、设置、诊断中的作者展示统一为“全部作者”语义，并解释逗号分隔多作者。 | 作者为空时不再显示“未设置作者”；多作者输入有清晰提示。 |
| BIZ-03 | P1 | 已完成 | 多作者/全部作者报告会把不同人的提交混在一起，团队周报不够自然。 | 周报、月报、自定义报告支持按作者分组展示，再在作者下按项目组织。 | 多作者或空作者报告中能看到每位作者的独立小节，并保留项目分组。 |
| BIZ-04 | P1 | 已完成 | 用户不确定当前报告是否“够交付”，尤其是提交太少、只有技术动作、缺少结果描述时。 | 增加报告质量提示：提交数、项目数、AI 状态、证据详情、可导出状态。 | 生成后能看到一组轻量质量提示，帮助用户决定是否补充说明或开启 AI。 |
| BIZ-05 | P1 | 已完成 | 项目名映射需要手填，用户第一次配置成本高。 | 根据仓库名、分支名和历史映射给出项目映射建议。 | 设置页能列出未映射仓库并提供一键填入建议名称。 |
| BIZ-06 | P1 | 已完成 | 用户常为不同场景写不同口径：日报、周会、绩效、项目复盘。 | 提供报告用途预设，驱动模板和 AI 指令的默认口径。 | 用户能选择“日报同步/周会汇报/绩效材料/项目复盘”等用途，并影响生成文本结构。 |
| BIZ-07 | P2 | 已完成 | 同一个人可能有多个 Git name/email，跨公司或多设备时报告容易漏。 | 增加作者身份别名管理，把多个 Git 作者合并为一个展示身份。 | 设置页可维护别名组，提取时按组匹配，报告展示统一姓名。 |
| BIZ-08 | P2 | 已完成 | 提交信息里的 issue、PR、需求编号没有被利用，报告缺少可追踪证据。 | 自动识别 commit message 中的 issue/PR/工单编号并生成证据链接或标签。 | 开启证据详情时，报告能展示识别出的编号，并在可配置前缀下生成链接。 |
| BIZ-09 | P2 | 已完成 | 历史报告只能按生成时间查看，复用和查找能力有限。 | 给历史报告增加筛选和搜索：类型、日期、项目、是否 AI、是否已导出。 | 最近报告较多时，用户能快速筛出某个月报或某项目相关报告。 |
| BIZ-10 | P2 | 已完成 | 用户关心隐私，但只有文档说明，本地处理的可信感还可以更直接。 | 在诊断或设置中展示本地数据边界：扫描目录、凭据存储、AI 调用开关。 | 用户能在应用内看到哪些数据留在本机，哪些仅在开启 AI 时发送。 |

## 当前执行顺序

1. BIZ-01：先解决空报告时用户不知道下一步怎么排查的问题。
2. BIZ-02：紧接着统一“全部作者/多作者”的业务语义，降低配置误解。
3. BIZ-03：再提升团队报告的阅读结构。
4. BIZ-04：补上报告交付前的质量提示。
5. BIZ-05：继续降低项目名映射的首次配置成本。
6. BIZ-06：提供报告用途预设，帮助用户按同步、周会、绩效、复盘切换口径。
7. BIZ-07：支持作者身份别名，减少跨设备、多邮箱提交漏统计。
8. BIZ-08：识别 issue、PR 与工单编号，让报告证据更可追踪。
9. BIZ-09：增加历史报告筛选和搜索，提升复用效率。
10. BIZ-10：展示本地数据边界，强化隐私可信感。

## 下一阶段判断

当前 BIZ-01 到 BIZ-10 已完成，GitPulse 的核心体验已经从“能生成报告”推进到“能指导用户生成更可信、可追踪、可复用的报告”。下一阶段建议从“报告生成器”继续升级为“个人/团队工作报告工作台”：允许补充 Git 之外的事实、让 AI 修改更可审计、让工作区健康状态更透明，并降低跨团队复用配置的成本。

## 下一阶段候选任务

| ID | 优先级 | 状态 | 用户问题 | 建议改进 | 验收标准 |
| --- | --- | --- | --- | --- | --- |
| BIZ-11 | P1 | 待做 | 很多真实工作没有对应 commit，例如会议、联调、上线验证、线上支持，报告只靠 Git 会漏掉关键贡献。 | 增加“补充事项/备注”能力，可按报告周期追加非 Git 工作项，并参与 AI 润色和历史记录。 | 用户生成报告前能添加补充事项，最终报告和历史记录都能保留这些内容。 |
| BIZ-12 | P1 | 待做 | AI 润色后用户担心内容被改写过度或出现无依据表述，但目前只能看最终文本。 | 增加 AI 润色对照和事实提示，突出新增、改写、删除内容，并提醒保留证据块。 | AI 润色后能看到原稿/润色稿差异，并能一键回退或接受。 |
| BIZ-13 | P1 | 待做 | 仓库移动、删除、分支变化后，用户不清楚当前索引是否过期。 | 增加工作区健康视图：上次扫描时间、失效仓库、禁用仓库、分支状态和快速重扫入口。 | 用户能在一个视图里发现失效仓库并执行重扫、禁用或移除。 |
| BIZ-14 | P2 | 待做 | 团队内经常复用同一套作者别名、项目映射、证据链接和模板，但目前需要手动复制配置。 | 增加配置方案导入/导出，支持共享工作区配置包。 | 用户能导出脱敏配置文件，并在另一台机器导入映射、模板和链接规则。 |
| BIZ-15 | P2 | 待做 | 报告历史已经能筛选，但还不能围绕一个项目持续回看阶段性成果。 | 增加项目维度回顾视图，按项目聚合历史报告和关键提交证据。 | 用户选择项目后，能看到该项目近期报告、提交数、证据编号和导出记录。 |

## 下一阶段建议顺序

1. BIZ-11：先补齐 Git 之外的真实工作事实，让报告内容更完整。
2. BIZ-12：再提高 AI 润色的可审计性，降低“AI 改坏报告”的心理成本。
3. BIZ-13：随后强化工作区健康状态，减少索引过期导致的隐性漏报。
4. BIZ-14：在配置逐渐复杂后，提供可共享、可迁移的配置方案。
5. BIZ-15：最后围绕项目做长期回顾，把历史报告沉淀成项目复盘资产。

## 代码审查后续任务

本组任务来自 2026-07-07 的整体代码审查与功能体验复盘。优先级判断聚焦“用户能否清楚知道当前生成范围、下一步能做什么、长期运行是否可靠”。

| ID | 优先级 | 状态 | 用户问题 | 建议改进 | 验收标准 |
| --- | --- | --- | --- | --- | --- |
| BIZ-16 | P0 | 已完成 | 用户生成报告前后不容易一眼确认当前口径，容易误判作者、仓库、分支或证据开关。 | 在主工作台增加当前生成范围条，展示作者、启用仓库、分支范围、证据状态和导出状态。 | 主界面无需打开设置即可看到当前生成口径；切换配置后范围条同步更新。 |
| BIZ-17 | P0 | 已完成 | 输出目录未配置时，导出按钮会消失，用户不知道如何把报告保存成文件。 | 报告生成后始终显示导出入口；未配置输出时显示“设置导出”并引导到输出设置。 | 有报告内容时始终能看到导出相关按钮；未配置输出时点击会打开设置并给出明确提示。 |
| BIZ-18 | P0 | 已完成 | 首次没有工作区或扫描到 0 个仓库时，用户不知道是路径不对、没有 Git 仓库，还是需要重新扫描。 | 优化无工作区/0 仓库空状态，给出添加目录、检查 `.git`、重新扫描等下一步。 | 空状态包含具体原因、下一步按钮和可验证的扫描状态。 |
| BIZ-19 | P1 | 待做 | 设置页里基础项和高级项混在一起，新用户第一次配置负担偏重。 | 将工作区、作者、输出目录作为基础配置；分支、过滤、证据链接、详细日志归为高级配置。 | 新用户完成基础配置所需阅读项减少；高级项仍可展开访问。 |
| BIZ-20 | P1 | 待做 | 多仓库场景下仓库列表缺少搜索、批量启用和批量禁用，筛选成本高。 | 在仓库索引面板增加搜索与批量操作。 | 用户能按名称/路径搜索仓库，并一键启用或禁用当前筛选结果。 |
| BIZ-21 | P1 | 待做 | 当前部分弹层和菜单仍依赖自定义交互，键盘与焦点体验可继续加固。 | 补齐弹层、菜单和分裂按钮的键盘关闭、焦点回收、ARIA 语义与外部点击关闭。 | 主要弹层可用键盘完整操作；关闭后焦点回到触发按钮；自动化用例覆盖关键路径。 |
| BIZ-22 | P1 | 待做 | `isBusy` 会把预览区替换成加载态，轻量操作时用户会失去当前报告上下文。 | 区分全屏阻塞任务和轻量任务，让复制、导出、AI 菜单等操作保留预览内容。 | 导出/复制等轻量动作不遮挡已有报告；长耗时提取仍显示进度。 |
| BIZ-23 | P1 | 待做 | 窄屏、高缩放和小高度窗口下，工具栏、弹层与报告预览仍可能拥挤。 | 做响应式与高缩放硬化，覆盖 320px 宽、200% 缩放、小高度桌面窗口。 | 关键控件不重叠、不溢出；Playwright 截图覆盖窄屏和小高度场景。 |
| BIZ-24 | P2 | 待做 | Codex OAuth 与常规 OpenAI/Anthropic 配置混在一起，容易被误认为稳定主路径。 | 将 Codex OAuth 标记为实验能力，并隔离说明适用场景、限制和回退方式。 | 设置页明确展示实验标记；默认用户不会误选为普通 AI 提供方。 |
| BIZ-25 | P2 | 待做 | 本地仓库扫描如果遇到符号链接、junction 或循环目录，长期运行风险较高。 | 加固 Rust 仓库扫描逻辑，记录已访问路径并避免链接循环。 | 构造 symlink/junction 循环目录时扫描能正常结束，并报告可理解的警告。 |
| BIZ-26 | P2 | 待做 | 最近报告存放在 localStorage，报告变多或内容较长时可能触发容量限制。 | 将报告历史迁移到 Tauri 本地文件存储，或增加容量保护和降级策略。 | 大报告历史不会导致设置保存失败；超限时能提示并保留最近可用记录。 |
| BIZ-27 | P2 | 待做 | 构建与测试已有覆盖，但缺少稳定的可访问性、窄屏和真实 Tauri smoke 门禁。 | 增加 a11y、窄屏/高缩放、Tauri smoke 的自动化验证。 | CI 或本地验证能覆盖主工作台生成、设置、导出和关键无障碍断言。 |

## 代码审查后续建议顺序

1. BIZ-16 + BIZ-17：先让主流程“看得懂、存得出”，这是最高频的信心问题。
2. BIZ-18 + BIZ-19：再降低首次配置和无结果状态的理解成本。
3. BIZ-20 + BIZ-21 + BIZ-23：随后完善多仓库、键盘访问和响应式硬化。
4. BIZ-22 + BIZ-24：优化操作状态模型，并隔离实验性 AI 登录路径。
5. BIZ-25 + BIZ-26 + BIZ-27：最后补长期可靠性与质量门禁。
