# GitPulse 业务体验改进路线图

## 背景

GitPulse 当前已经能完成多仓库扫描、日报/周报/自定义区间/月报生成、AI 润色、导出和诊断。下一阶段的业务重点不应只是增加按钮，而是减少用户在真实交付报告时的判断成本：为什么没有结果、报告够不够能交付、多人/多身份怎么汇总、项目名怎样更贴近业务语境。

## 优先级说明

- P0：高频阻塞，直接影响用户能否顺利生成报告。
- P1：明显提升报告可交付性或团队场景覆盖。
- P2：增强专业度、复用性或长期留存。

## 值得推进的业务任务

| ID | 优先级 | 状态 | 用户问题 | 建议改进 | 验收标准 |
| --- | --- | --- | --- | --- | --- |
| BIZ-01 | P0 | 已完成 | 生成空报告时，用户不知道是没有提交、作者不匹配、仓库没扫到，还是时间范围选错。 | 在空报告和警告区域给出可行动原因与下一步建议。 | 当提交数为 0 时，界面能说明当前筛选范围，并提示检查作者、日期、分支范围或仓库索引。 |
| BIZ-02 | P0 | 已完成 | 空作者现在代表“全部作者”，但界面仍容易让用户误解为配置缺失。 | 将主界面、设置、诊断中的作者展示统一为“全部作者”语义，并解释逗号分隔多作者。 | 作者为空时不再显示“未设置作者”；多作者输入有清晰提示。 |
| BIZ-03 | P1 | 待做 | 多作者/全部作者报告会把不同人的提交混在一起，团队周报不够自然。 | 周报、月报、自定义报告支持按作者分组展示，再在作者下按项目组织。 | 多作者或空作者报告中能看到每位作者的独立小节，并保留项目分组。 |
| BIZ-04 | P1 | 待做 | 用户不确定当前报告是否“够交付”，尤其是提交太少、只有技术动作、缺少结果描述时。 | 增加报告质量提示：提交数、项目数、AI 状态、证据详情、可导出状态。 | 生成后能看到一组轻量质量提示，帮助用户决定是否补充说明或开启 AI。 |
| BIZ-05 | P1 | 待做 | 项目名映射需要手填，用户第一次配置成本高。 | 根据仓库名、分支名和历史映射给出项目映射建议。 | 设置页能列出未映射仓库并提供一键填入建议名称。 |
| BIZ-06 | P1 | 待做 | 用户常为不同场景写不同口径：日报、周会、绩效、项目复盘。 | 提供报告用途预设，驱动模板和 AI 指令的默认口径。 | 用户能选择“日报同步/周会汇报/绩效材料/项目复盘”等用途，并影响生成文本结构。 |
| BIZ-07 | P2 | 待做 | 同一个人可能有多个 Git name/email，跨公司或多设备时报告容易漏。 | 增加作者身份别名管理，把多个 Git 作者合并为一个展示身份。 | 设置页可维护别名组，提取时按组匹配，报告展示统一姓名。 |
| BIZ-08 | P2 | 待做 | 提交信息里的 issue、PR、需求编号没有被利用，报告缺少可追踪证据。 | 自动识别 commit message 中的 issue/PR/工单编号并生成证据链接或标签。 | 开启证据详情时，报告能展示识别出的编号，并在可配置前缀下生成链接。 |
| BIZ-09 | P2 | 待做 | 历史报告只能按生成时间查看，复用和查找能力有限。 | 给历史报告增加筛选和搜索：类型、日期、项目、是否 AI、是否已导出。 | 最近报告较多时，用户能快速筛出某个月报或某项目相关报告。 |
| BIZ-10 | P2 | 待做 | 用户关心隐私，但只有文档说明，本地处理的可信感还可以更直接。 | 在诊断或设置中展示本地数据边界：扫描目录、凭据存储、AI 调用开关。 | 用户能在应用内看到哪些数据留在本机，哪些仅在开启 AI 时发送。 |

## 当前执行顺序

1. BIZ-01：先解决空报告时用户不知道下一步怎么排查的问题。
2. BIZ-02：紧接着统一“全部作者/多作者”的业务语义，降低配置误解。
3. BIZ-03：再提升团队报告的阅读结构。
4. BIZ-04：补上报告交付前的质量提示。

## 下一阶段判断

当 BIZ-01 到 BIZ-04 完成后，GitPulse 的核心体验会从“能生成报告”推进到“能指导用户生成更可交付的报告”。下一阶段可以围绕两个方向继续：一是降低配置成本（BIZ-05、BIZ-07），二是提升报告证据和复用价值（BIZ-08、BIZ-09）。
